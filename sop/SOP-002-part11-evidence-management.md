## 1. Purpose

This SOP establishes the procedures for creating, maintaining, and verifying electronic records and evidence chains in the Solace Browser platform. The system is **Part 11 Architected** — the architecture is designed to support FDA 21 CFR Part 11 requirements. Validation and formal compliance certification is the responsibility of each customer's quality organization.

### 1.1 Important Distinction

| Term | Meaning |
|------|---------|
| **Part 11 Architected** | The system's architecture supports Part 11 requirements by design |
| **Part 11 Compliant** | The customer has validated the system per their own validation protocol (IQ/OQ/PQ) |
| **Part 11 Certified** | NOT CLAIMED — certification requires third-party auditor engagement |

## 2. Scope

Applies to all electronic records generated by:
- Solace Browser automation runs (recipe execution evidence)
- OAuth3 token lifecycle events (grant, revoke, step-up)
- Evidence bundle creation, sealing, and storage
- Audit trail entries in `evidence_chain.jsonl`

## 3. ALCOA+ Principles Enforcement

Every evidence record MUST satisfy all ALCOA+ properties:

| Property | Implementation | Verification |
|----------|---------------|-------------|
| **A**ttributable | `token_id` + `subject` on every entry | Mandatory field validation |
| **L**egible | `action_type` string (human-readable) | Schema validation |
| **C**ontemporaneous | `unix_timestamp` captured at write time | System clock, not user input |
| **O**riginal | `input_hash` + `output_hash` (SHA-256) | Hash verification on read |
| **A**ccurate | `result` (pass/fail/code) mandatory | No null results allowed |
| **C**omplete | `scope_verified` + `session_id` mandatory | Missing fields = BLOCKED |
| **C**onsistent | Hash chain integrity verified on seal | Chain break = ERROR |
| **E**nduring | `chmod 444` after seal, 90-day retention (Pro) | File permissions verified |
| **A**vailable | PZip compression + cloud sync (Pro tier) | Retrieval test on upload |

## 4. Evidence Chain Procedure

### 4.1 Chain Creation

```
1. Genesis entry created with prev_hash = 0x000...0
2. Each subsequent entry includes:
   - action_hash = SHA-256(action_type + input_hash + output_hash + timestamp)
   - chain_hash = SHA-256(previous_chain_hash + action_hash)
3. Chain is append-only — no modifications after write
```

### 4.2 Chain Integrity Verification

```
For each entry E[n] in the chain:
1. Compute action_hash from E[n] fields
2. Compute expected_chain_hash = SHA-256(E[n-1].chain_hash + action_hash)
3. Compare expected_chain_hash with E[n].chain_hash
4. If mismatch: EVIDENCE_INTEGRITY_ERROR (chain is broken)
5. If match: continue to E[n+1]
```

### 4.3 Bundle Sealing

```
1. Compute bundle_hash = SHA-256(all files in bundle)
2. Write bundle_hash to bundle.sha256
3. Set file permissions: chmod 444 (read-only)
4. Record seal event in evidence_chain.jsonl
5. Upload to cloud (if Pro tier, after seal only)
```

## 5. Electronic Signatures

### 5.1 User Approval (PREVIEW_READY → APPROVED)

Every user approval constitutes an electronic signature and MUST include:

| Field | Requirement |
|-------|------------|
| `user_id` | Non-empty string identifying the signer |
| `meaning` | Non-empty string explaining why they approved |
| `timestamp` | ISO 8601 UTC timestamp |
| `run_id` | Unique identifier for the execution run |
| `preview_hash` | SHA-256 of the preview content they reviewed |

### 5.2 Anti-Clippy Law (Electronic Signature Protection)

The following are FORBIDDEN and enforced by code:
- Auto-approval (no programmatic approval without human action)
- Empty user_id (who signed?)
- Empty meaning (why did they sign?)
- Approval of non-PREVIEW_READY state (signing something that wasn't reviewed)

## 9. Incident Management

### 9.1 Evidence Chain Integrity Failure

If a hash chain break is detected:

1. **HALT** all operations on the affected chain
2. **DOCUMENT** the exact point of failure (entry index, expected vs actual hash)
3. **PRESERVE** the corrupted chain (do NOT delete)
4. **NOTIFY** the system administrator and data integrity team
5. **INVESTIGATE** root cause (tampering, corruption, software bug)
6. **REMEDIATE** based on root cause analysis
7. **DOCUMENT** the incident and resolution in the incident log

### 9.2 This Is NOT a Fallback

The system does NOT silently continue when evidence integrity fails. Per the Fallback Ban (Paper 08), evidence integrity errors HALT the system. This is by design.
