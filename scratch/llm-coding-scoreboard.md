## PROCESS ASSESSMENT: /llm-coding vs Regular Coding

### The Experiment

**Task:** Turn a brainstorm document (yinyang-sidebar-rethink.md) into a production-ready
architecture spec using 3 LLMs as a review panel.

### Before (Regular Coding Process)

| Dimension | Score | Notes |
|-----------|-------|-------|
| Architecture quality | 4/10 | Solo dev reviews own work — blind spots everywhere |
| Security coverage | 3/10 | "I'll add auth later" — famous last words |
| Business model validation | 2/10 | No external challenge to pricing/moat assumptions |
| Implementation realism | 3/10 | Timeline fantasies go unchecked |
| Competitive accuracy | 4/10 | Claims not verified against current competitor features |
| Iteration speed | 7/10 | Fast but in circles (no external signal to course-correct) |
| Cost per review cycle | 8/10 | Free (just your own time, but time is expensive) |
| Total | **31/70** | **44%** |

### After (LLM-Coding Process)

| Dimension | Score | Notes |
|-----------|-------|-------|
| Architecture quality | 9/10 | 3 independent architects catch different issues |
| Security coverage | 9/10 | All 3 LLMs found critical CDP/auth/token issues I missed |
| Business model validation | 8/10 | Enterprise pricing bug caught, free tier too stingy caught |
| Implementation realism | 8/10 | 6-week fantasy → 12-week reality check (unanimous) |
| Competitive accuracy | 8/10 | Claims verified: Tauri size, storage_state limits, MV3 gaps |
| Iteration speed | 9/10 | 3 parallel reviews in ~3 min vs ~18 min sequential |
| Cost per review cycle | 10/10 | $0.00 (web LLMs, included in subscription) |
| Total | **61/70** | **87%** |

### Delta: +30 points (44% → 87%) — 2x improvement

### Key Insight

**The 3-LLM panel found 42 unique issues across 3 rounds.** I would have found maybe 10-15
of those on my own. The remaining 27-32 are the kind of issues that only surface in
production — except we found them in a markdown file before writing a single line of code.

**Most impactful catches:**
1. Token bootstrap impossible via chrome.storage.session (would have been a week of debugging)
2. SW keep-alive hack breaks on Chrome 127+ (would have shipped broken)
3. Enterprise pricing cheaper per-seat than Team (would have lost money)
4. CDP raw proxy = RCE vulnerability (would have shipped a security hole)
5. PyInstaller distribution needed (would have hit this at launch day)

### Verdict

/llm-coding is NOT a replacement for coding. It's a replacement for **architecture review**.
It compresses 3 expert reviews into 3 minutes at $0 cost. The doc went from a brainstorm
(72/100) to a production-ready spec (86/100) in 3 rounds, ~45 minutes total.

**When to use /llm-coding:** Before writing code. Any architecture doc, design proposal,
or spec that would normally go to a senior architect for review.

**When NOT to use it:** For actual code review (use /notebook-qa + /notebook-coding instead).
