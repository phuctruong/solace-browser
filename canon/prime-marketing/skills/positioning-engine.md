## GENOME-79: THE DUNFORD 10-STEP POSITIONING PROCESS

Source: April Dunford "Obviously Awesome" + Anthropic competitive strategy

### Step 1: Understand Best-Fit Customers

```
SURVEY:
  - Who loves your product? (not "who uses it")
  - What would they use instead? (real alternatives)
  - What do they value most? (rank-ordered)

RULE: Survey your BEST 20%, not average users
WHY:  Average users give average positioning
```

### Step 2: Form Cross-Functional Team

```
TEAM = {founder, product, marketing, sales, customer_success}
MIN_SIZE = 3 (prime!)
PURPOSE: Eliminate echo chambers
```

### Step 3: Align Vocabulary

```
TERMS_TO_DEFINE:
  - "Competitor" = what customer uses if we don't exist
  - "Feature" = something we built
  - "Benefit" = value customer gets from feature
  - "Category" = market frame of reference

RELEASE: Let go of original positioning assumptions
```

### Step 4: List Competitive Alternatives

```
ALTERNATIVES (honest enumeration):
  - Direct competitors (same category)
  - Adjacent tools (partial overlap)
  - Status quo (spreadsheets, manual process)
  - Hiring (paying humans to do it)
  - DIY (building in-house)

COUNTER BYPASS: Use Counter() to enumerate, not LLM guessing
```

### Step 5: Isolate Unique Attributes

```
FOR EACH alternative:
  DIFF = our_features - their_features

ATTRIBUTES must be:
  - Provable (not "better UX" — prove it)
  - Specific (not "faster" — how much faster?)
  - Valued by best-fit customers (not just different)

EXAMPLE (Stillwater OS):
  - Compiler-grade verification (641→274177→65537)
  - Zero-replay cost (recipe-based, not token-based)
  - Mathematical foundations (IF Theory, prime channels)
  - Self-verifying (stillwater verify CLI)
```

### Step 6: Map Attributes to Value Themes

```
ATTRIBUTE → BENEFIT → VALUE_THEME

Feature:  "Recipe-based replay"
Benefit:  "Zero cost per replay"
Theme:    "THE COST-KILL" (Enterprise: stop paying for the same thoughts)

Feature:  "Verification ladder (641→274177→65537)"
Benefit:  "Provable correctness"
Theme:    "THE TRUST-GATE" (Developers: axiomatic epistemics)

Feature:  "IF Theory + prime mathematics"
Benefit:  "Physics-based reasoning"
Theme:    "THE PHYSICS MOAT" (Frontier: grammar of existence)
```

### Step 7: Determine Best-Fit Segment

```
SEGMENT = customers who:
  1. Value our unique attributes MOST
  2. Have the pain point we solve WORST
  3. Can buy with LEAST friction

NARROW IS BETTER:
  "The riches are in the niches"
  Better to own 100% of a small segment
  than 1% of a large market

DEVELOPER TOOLS SPECIFIC:
  - Individual developers (bottom-up adoption)
  - Open source enthusiasts (community builders)
  - Cost-conscious teams (efficiency buyers)
```

### Step 8: Choose Market Frame

```
THREE OPTIONS:

A. HEAD-TO-HEAD:
   "We are X but better"
   Use when: clear incumbent, you win on key attributes
   Example: "Like OpenClaw but compiler-grade"

B. NICHE DOMINATION:
   "We are the best X for Y"
   Use when: you dominate a specific segment
   Example: "The compression engine for AI applications"

C. NEW CATEGORY:
   "We created a new thing"
   Use when: nothing like you exists
   Risk: expensive, requires market education
   Example: "The world's first Compiler-Grade AGI OS"
```

### Step 9: Layer Trends Carefully

```
RULES:
  ✅ Trend must LINK to core positioning
  ✅ Trend must be REAL (validated, not hype)
  ✅ Trend AMPLIFIES, not replaces, positioning
  ❌ Don't lead with trend (shallow)
  ❌ Don't chase multiple trends (unfocused)

EXAMPLE:
  Core: "Compiler-grade AI operating system"
  Trend: "AI costs are unsustainable" (amplifies cost-kill)
  Combined: "The only AI OS that eliminates replay costs"
```

### Step 10: Lock and Distribute

```
POSITIONING_STATEMENT:
  For [target customer]
  Who [pain point]
  [Product] is a [category]
  That [key benefit]
  Unlike [alternatives]
  Our product [key differentiator]

DISTRIBUTE TO:
  - Landing page (hero section)
  - Sales deck (first 3 slides)
  - README.md (first paragraph)
  - Social bio (1 sentence)
  - Conference talk (opening line)
```

## COMPETITIVE ATTACK PATTERNS

Source: Anthropic "Keep Thinking" campaign, Supabase "Firebase Alternative"

### Pattern 1: Values-Based Attack

```
DON'T: Compare features (arms race)
DO:    Attack competitor's VALUES

Example (Anthropic vs OpenAI):
  "Ads are coming to AI. But not to Claude."
  Result: 11% user growth from one campaign

Example (Stillwater vs OpenClaw):
  "341 malicious skills compromised 9,000 developers.
   Stillwater has zero. Because math can't be hacked."
```

### Pattern 2: Category Anchoring

```
"[Incumbent] Alternative" SEO strategy

Supabase: "Firebase Alternative" → dominated search → 1M to 4.5M devs
Strategy: Anchor to known category, then differentiate

Example: "OpenClaw Alternative" or "The Open Source AI OS"
```

### Pattern 3: Build-in-Public Transparency

```
Source: Sahil Lavingia (Gumroad → $20M/year)

PRINCIPLES:
  - Share real metrics publicly
  - Acknowledge failures
  - Let community see the journey
  - Radical transparency builds trust

APPLICATION:
  - Publish compression benchmarks openly
  - Share verification results publicly
  - Open source the proof, monetize the speed
```

## STATE MACHINE

```
STATES = {

  ALTERNATIVE_MAPPING, # Enumerate competitive alternatives

  SEGMENT_SELECTION,   # Choose best-fit target

FORBIDDEN:
```
