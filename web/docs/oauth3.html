OAuth3 Safety

What is OAuth3?

OAuth3 is Solace Browser's agency delegation standard. It extends OAuth 2.0 with three missing properties that matter for AI agents:

  • Scoped authority — each token grants exactly the permissions it names, nothing more
  • Time-bounded — every token has an expiry, after which it cannot be used
  • Budget-gated — every action checks a budget before executing (action count + spend caps)

OAuth3 means: your AI agent can only do what you explicitly approved, within the limits you set, for as long as you allow.

The approval gate

Before any sensitive action, Solace Browser shows the OAuth3 confirmation dialog. This dialog displays:

  • The app and action being requested
  • The OAuth3 scopes (what permissions are being used)
  • Your remaining budget for this action type
  • Three safety guarantees (time-bounded, revocable, evidence-sealed)

You have 15 seconds to approve or reject. If you do nothing, the action is automatically cancelled (fail-closed). There is no auto-approve, ever.

Budget enforcement

Every app has two budget planes:

Action budgets (non-financial)

Controls what actions the app can take and how many:

Gmail: {
  sends_per_day: 10,
  deletes_per_day: 3,
  max_recipients_per_email: 3,
  reads_per_run: 50
}

LinkedIn: {
  posts_per_day: 1,
  comments_per_day: 5,
  connection_requests_per_day: 10
}

Spend budgets (financial)

Controls LLM costs and payment authority:

{
  llm_per_day_cents: 50,     // $0.50/day max on AI
  llm_per_run_cents: 10,     // $0.10/run max
  per_transaction_max: 200   // $200 max per payment
}

The 5 budget gates

Every action passes through 5 gates. Any failure = blocked:

  • B1 Policy Present — scope requires policy → policy must exist
  • B2 Limit Remaining — action count must not exceed remaining budget
  • B3 Target Allowed — target must be on your allowlist
  • B4 Step-Up Required — if re-consent required → pauses for approval
  • B5 Evidence Mode — if policy requires evidence → must comply

Budget resolution: when global AND app budgets conflict, the stricter rule wins. Allowlists use intersection (smaller set). Step-up: if any policy requires it, it's required.

Evidence trails

Every approved action is sealed to your evidence vault:

  • SHA-256 hash of the action + inputs + outputs
  • Timestamp and user approval record
  • Sequential hash chain (tamper-evident)
  • Part 11 compliant — audit trail cannot be edited

Evidence is stored locally in ~/.solace/evidence/. With a Solace Pro account, it syncs to encrypted cloud storage (30-day retention).

Fail-closed model

Solace Browser is fail-closed: when in doubt, it does nothing.

  • No approval = no action — the 15-second countdown auto-cancels
  • No credits = no LLM call — if your credit balance is zero, execution stops
  • No scope = no token — every action validates OAuth3 scope at runtime
  • No seal = no evidence — every run creates a hash chain or it doesn't complete
  • No consent = no billing — every charge requires a prior user action

This is the core invariant. It cannot be disabled by configuration.

Revocation

You can revoke any OAuth3 token at any time from Settings → Account. Revoked tokens cannot be used, even if they haven't expired. All active sessions using that token are terminated immediately.

To adjust budgets: Settings → Apps → select app → Budget. Changes take effect on the next run.

FAQ

Can an agent approve actions on my behalf?

No. The OAuth3 approval gate always requires a human action (clicking [Approve] in the UI). Agents cannot simulate this click — the UI checks that the approval came from the real browser interface, not from the API.

What if I close the browser during a run?

The run is cancelled. No partial actions are committed. The evidence vault records "run cancelled" with the reason.

Can I pre-approve a category of actions?

Yes — via recipe pinning. If you pin a recipe and set auto-approve for read-only actions, those specific actions in that recipe can proceed without manual approval. Write actions (send, delete, post) always require manual approval.

Is OAuth3 the same as OAuth 2.0?

OAuth3 builds on OAuth 2.0 patterns but adds budget enforcement, evidence sealing, and the fail-closed model. It is not a replacement for OAuth 2.0 — it is a higher-level agency delegation standard.
